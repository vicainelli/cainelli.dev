# {frontmatter.title}

> "Do you have experience with Rust?"
> "Not yet, but I can definitely figure it out."

If you work in tech, you have probably lived this moment.

Someone asks about a language or framework you have never used. You can say "no" and freeze, or you can say "not yet" and start learning.

Most days in software are exactly that. We solve new problems with a mix of old lessons, fast research, and a bit of trial and error.

## The skill that actually matters

The stack changes all the time. What stays valuable is your ability to get useful quickly in unfamiliar territory.

I have seen engineers with "perfect" resumes struggle when the context shifts. I have also seen curious generalists become indispensable in a few weeks because they ask good questions and learn in public.

## Learning agility is the new technical skill

Learning agility is not "read docs faster." It is knowing how to break a problem down, test assumptions, and recover when your first approach fails.

Specific tools expire. The ability to figure things out does not.

## How to actually develop this

The fastest way I know is to take projects that scare you a little.

Pick work that forces you to learn one important new thing. Pair with someone stronger in that area. Ship something small first, then improve it.

It is not about pretending to know everything. It is about staying calm while you learn.

## Trust your process

Next time someone asks if you know a technology you have never touched, "not yet" is a valid answer.

Your value is not only what you already know. It is how reliably you can learn what comes next.

That is the part I still love about this job: we build things while we are still learning how to build them.
